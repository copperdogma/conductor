## Focus

Explain and route the recurring new-worktree setup failure mode where agents
hit messages like `vitest: command not found`, discover `node_modules` is
absent, and then decide ad hoc whether to run `npm ci`, `pnpm install`, or a
Python environment restore.

Surfaces compared:

- `projects.yaml` comparison surfaces for `.codex/environments/environment.toml`,
  local runtime launchers, setup checklists, and setup runbooks
- Alignment 016 local dev port contract
- Alignment 032 portfolio local runtime allocation
- `.codex/environments/environment.toml` in Dossier, Storybook, CineForge, and
  Echo Forge
- setup-methodology local runtime guidance across tracked repos
- target repo package/lock and local runtime surfaces
- current target primary checkout status before any recommendation

| Echo Forge | present | empty | `npm run local:*` actions for app, review, auditor, studio |

This means Codex can show useful Run actions for some repos, but a fresh
worktree still has no repo-owned dependency setup path. When an agent sees a
missing test binary, it is not necessarily using Codex wrong. The repo simply
has no non-empty Codex setup command for the app to run or for the agent to
trust.

Primary checkouts sometimes hide the gap because dependencies already exist
there. During this read-only pass, primary checkouts showed Dossier with a
`.venv`, Storybook/CineForge/Echo Forge with `node_modules`, and several repos
without local install artifacts. That does not prove new worktrees are ready;
new worktrees normally need their own dependency artifacts.

Alignment 016 solved fixed app port collisions. Alignment 032 solved
portfolio-level runtime allocation and status/stop behavior. Neither alignment
defined dependency hydration for new Codex worktrees.

## Classification

- **Portable improvement:** each active code repo should expose one repo-owned,
  idempotent setup command for Codex worktrees. The Codex environment file
  should call that command instead of leaving `[setup].script` empty or
  embedding long package-manager logic directly in TOML.
- **Portable improvement:** setup commands should be lockfile-respecting,
  repeatable, and explicit about what they install. They should restore ignored
  dependency artifacts only, not rewrite source files, lockfiles, generated
  methodology surfaces, or user data.
- **Portable improvement:** runtime-heavy repos should keep Run/status actions,
  but those actions should sit behind the same setup contract so local launch
  failures do not masquerade as code bugs.
- **Intentional adaptation:** repos without a local browser/API runtime do not
  need Run actions yet. They may still need a setup script when their validation
  commands depend on local package installs.
- **Intentional adaptation:** setup commands should stay repo-local because
  Python, pnpm, npm, nested UI packages, pinned runtimes, and external provider
  credentials differ materially across projects.
- **Unclear drift:** some repos have `.codex/environments/environment.toml` and
  some do not. The environment files are marked autogenerated, but this pass did
  not find a Conductor-owned generator or setup contract that explains which
  repos should have the file.

## Human Judgment

The open policy question is not whether agents should run `npm ci` when a tool
is missing. They should stop guessing. The useful decision is whether Codex
worktrees should have an explicit repo-owned setup hook that may install local
dependencies when invoked.

Recommendation: yes, but keep it explicit and idempotent. Do not run background
installs outside the Codex setup hook, do not mutate source or lockfiles, and do
not add Run actions for repos without real runtime surfaces.

## Practical Impact

Cam should see fewer setup narrations and fewer false blockers like missing
`vitest` in fresh worktrees. Agents get one known setup command per repo, so
dependency restoration becomes routine validation prep instead of a fresh
judgment call every time.
